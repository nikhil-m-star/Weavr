# Design Decisions

### Email Re-Verification Rule
In a Clerk-authenticated workflow, the user's email verification and change management are handled directly by Clerk. When a student updates their email address inside Clerk, Clerk requires immediate OTP confirmation before updating the user context. In our application profile database, when the student completes an email change, their `verified` status remains active for historical submissions. However, to comply with re-verification constraints, we enforce that new applications are blocked at the API layer until they have completed the Clerk OTP confirmation flow. This ensures that historical applications stay valid and active, while new application submissions remain restricted until the email change is finalized.

### Concurrency Cap Race Conditions & Withdrawal Reopening
To prevent double-accept race conditions at the listing capacity cap, applications are processed inside a Prisma transaction utilizing database-level row locking. We issue a `SELECT ... FOR UPDATE` query on the `Listing` row, blocking other concurrent applications until the current transaction completes. If the application cap is reached, the transaction automatically flips the listing's status to `CLOSED` and stamps it with an `AUTO_CAP` reason. Conversely, when an applicant withdraws (which is only allowed while the application is in the `SUBMITTED` state), the system decrements the denormalized count. If the listing was previously auto-closed (indicated by `closedReason: 'AUTO_CAP'`) and the count drops below the limit, the system automatically transitions the status back to `ACTIVE`. This is a system-triggered reversal of a system-triggered action, and is explicitly allowed because it is deterministic and does not bypass manual business workflows.

### Skill Updates & Match Score Computational Complexity
When a company edits an `ACTIVE` listing's required or preferred skills, we do not perform retroactive batch calculations in the database. Instead, matching scores are treated as highly dynamic, read-time calculations that are never stored in the database. When a student queries `GET /api/listings`, we optimize the retrieval by pre-fetching the student's skills and query all active listings with their skills in a single database query. We then compute the scores (required skill overlap (40%), preferred skill overlap (20%), branch/gradYear alignment (15%), completeness (15%), and exponential recency decay (10%)) in-memory in a single pass. This ensures $O(n)$ time complexity relative to the number of active listings and completely avoids $O(n^2)$ nested database queries, keeping the load light even with multiple simultaneous users.
